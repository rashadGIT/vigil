# Requirements: Vigil

**Defined:** 2026-04-05
**Core Value:** Never miss a step in a service again

## v1 Requirements (Phase 1 — Funeral Operations System)

### Authentication & Multi-Tenancy

- [ ] **AUTH-01**: Staff can log in with email + password via AWS Cognito
- [ ] **AUTH-02**: JWT carries `tenantId` and `role`; every protected route validates both
- [ ] **AUTH-03**: `DEV_AUTH_BYPASS=true` allows fully offline local dev without AWS credentials
- [ ] **AUTH-04**: `cognitoSub` synced to User record on every real login
- [ ] **AUTH-05**: Every database table has `tenantId`; Prisma `forTenant()` extension auto-injects filter on all queries
- [ ] **AUTH-06**: Staff can be invited and assigned roles (admin | staff)

### Case Management

- [ ] **CASE-01**: Staff can create a case with deceased info, service type, and assigned staff
- [ ] **CASE-02**: Case dashboard lists all cases with status, assigned staff, and overdue task indicators
- [ ] **CASE-03**: Case status transitions: new → in_progress → completed → archived
- [ ] **CASE-04**: Per-case workspace shows tabbed view: info, contacts, tasks, documents, payments, signatures, vendors
- [ ] **CASE-05**: Case timeline / audit log records every write action (create, update, delete, view_sensitive)
- [ ] **CASE-06**: Soft delete: 90-day recoverable → 7-year archived → hard delete

### Intake

- [ ] **INTK-01**: Public intake form hosted at `[slug].vigilhq.com/intake` and embeddable on client website
- [ ] **INTK-02**: Intake submission atomically creates: case + family contact + templated task list + follow-up schedule
- [ ] **INTK-03**: Intake form is mobile-first (designed for 2am phone use)

### Tasks

- [ ] **TASK-01**: Task checklist generated from service type template (burial, cremation, graveside, memorial)
- [ ] **TASK-02**: Staff can check off tasks, assign to team members, and set due dates
- [ ] **TASK-03**: Overdue tasks surface on case dashboard
- [ ] **TASK-04**: Admin can customize task templates per service type

### Obituary

- [ ] **OBIT-01**: System auto-generates obituary draft by interpolating intake data into editable template
- [ ] **OBIT-02**: Staff can edit and approve the draft; approved text is stored separately from draft

### Documents

- [ ] **DOCS-01**: Staff can upload documents (death cert, service program, invoice, other) directly to S3 via presigned PUT URL
- [ ] **DOCS-02**: Documents are accessible via 15-minute signed GET URLs; buckets are private
- [ ] **DOCS-03**: Documents are attached to a case and visible in the documents tab

### Payments

- [ ] **PAY-01**: Staff can record payment amounts, method, and notes per case
- [ ] **PAY-02**: Case workspace shows total amount, amount paid, and outstanding balance
- [ ] **PAY-03**: No raw card numbers stored — platform records payments only (processing is external)

### E-Signatures

- [ ] **SIGN-01**: Staff can send a signature request link to a family contact
- [ ] **SIGN-02**: Family completes signature on any device at `/sign/[token]` (public, no auth)
- [ ] **SIGN-03**: Signed document stored with full ESIGN/UETA audit trail: signer name, email, IP, timestamp, document hash
- [ ] **SIGN-04**: Signed copies visible in case workspace permanently

### FTC GPL Compliance

- [ ] **GPL-01**: Admin can manage a price list (categories: professional services, facilities, vehicles, merchandise)
- [ ] **GPL-02**: Staff can select services on a case, generating case line items with prices
- [ ] **GPL-03**: System generates a PDF General Price List from selected services (pdfkit, outputs to S3)

### Grief Follow-Up Automation

- [ ] **FLWP-01**: n8n Workflow 1 auto-schedules follow-up emails at 1 week, 1 month, 6 months, 1 year after service completion
- [ ] **FLWP-02**: Follow-up emails are personalized with family name and funeral home branding
- [ ] **FLWP-03**: Follow-up status (pending | sent | failed) is visible per case
- [ ] **FLWP-04**: `CronStubsService` provides offline substitute for local dev (logs counts, no sends)

### Vendor Coordination

- [ ] **VEND-01**: Admin can manage a vendor directory (florist, clergy, musician, caterer, vault, livery, crematory, other)
- [ ] **VEND-02**: Staff can assign vendors to a case with role and status (requested | confirmed | completed)
- [ ] **VEND-03**: n8n Workflow 2 sends staff notification when vendor is assigned to a case

### Calendar & Scheduling

- [ ] **CAL-01**: Staff can create calendar events (visitation, service, committal, pickup, preparation, meeting)
- [ ] **CAL-02**: Events can be linked to a case or standalone
- [ ] **CAL-03**: Staff can be assigned to events (join table, not array)
- [ ] **CAL-04**: Calendar UI shows monthly/weekly view with case color-coding

### Document Generation

- [ ] **DGEN-01**: System generates service program PDF from case data (pdfkit)
- [ ] **DGEN-02**: Generated documents are stored to S3 and attached to the case

### Settings & Branding

- [ ] **SETT-01**: Admin can update funeral home name, logo (S3 upload), and brand colors
- [ ] **SETT-02**: Admin can manage staff accounts (invite, assign roles, deactivate)
- [ ] **SETT-03**: Admin can customize task checklist templates per service type
- [ ] **SETT-04**: Tenant has `googleReviewUrl` field for review generation links

### Infrastructure & DevOps

- [ ] **INFR-01**: All AWS infrastructure provisioned via CDK TypeScript (7 stacks: foundation, network, auth, data, compute, amplify, observability)
- [ ] **INFR-02**: Next.js frontend deployed to AWS Amplify with wildcard `*.vigilhq.com` subdomain routing
- [ ] **INFR-03**: NestJS backend deployed to ECS Fargate in public subnets with `assignPublicIp: true`; ALB in front
- [ ] **INFR-04**: RDS PostgreSQL (db.t4g.micro) in isolated subnets
- [ ] **INFR-05**: ElastiCache Redis (cache.t4g.micro) in isolated subnets
- [ ] **INFR-06**: All secrets in AWS Secrets Manager; never in committed `.env` files
- [ ] **INFR-07**: CI/CD: GitHub Actions `ci.yml` (type-check, lint, build) + `security.yml` (npm audit, TruffleHog) + manual approval gate to production
- [ ] **INFR-08**: Monitoring: CloudWatch Logs + Sentry + BetterUptime + RDS Performance Insights
- [ ] **INFR-09**: Local dev runs fully offline via Docker Compose (Postgres + Redis) with `DEV_AUTH_BYPASS=true`

### Testing

- [ ] **TEST-01**: Unit tests (Jest) — pure functions, utilities, business logic — 80%+ coverage target
- [ ] **TEST-02**: Component tests (Jest + RTL) — UI rendering, user interactions
- [x] **TEST-03**: Contract tests (Jest, `@jest-environment node`) — API route handlers
- [ ] **TEST-04**: Integration test — tenant isolation verified (Tenant A cannot read Tenant B data)
- [x] **TEST-05**: E2E tests (Playwright) — intake → case created, login → dashboard, case workspace navigation
- [ ] **TEST-06**: `npm run test` and `npm run test:e2e` runnable in CI without manual setup

### Seed Data

- [ ] **SEED-01**: `prisma/seed.ts` creates 2 demo tenants (Sunrise Standard, Heritage Pilot)
- [ ] **SEED-02**: Seed creates 3 Sunrise cases in different states (new, in_progress, completed) with realistic names
- [ ] **SEED-03**: Seed creates Cognito users when AWS creds present; falls back to stub `cognitoSub` values offline
- [ ] **SEED-04**: Demo environment runnable with `npx prisma db seed` — matches 15-minute demo script

## v2 Requirements (Phase 2 — after 2-3 clients)

### Family Portal
- **FPRT-01**: Family receives secure access link to view case details
- **FPRT-02**: Family can upload photos and approve obituary draft
- **FPRT-03**: Family can view payment status

### Decedent Tracking
- **DTRK-01**: Chain-of-custody status board: pending_pickup → in_transit → received → in_preparation → ready → at_service → released
- **DTRK-02**: All staff see current location/status in real time

### Referral Tracking
- **RREF-01**: Staff can tag the source of each case (hospice, hospital, nursing home, Google, referral, walk-in, other)
- **RREF-02**: Admin can view referral source breakdown per period

### Mobile PWA
- **PWA-01**: App installable from browser (manifest.json, service worker)
- **PWA-02**: Case list and task checklist readable offline

### Pre-planning Portal
- **PPLN-01**: Public-facing form for families to submit pre-need plans
- **PPLN-02**: Submissions auto-route to staff as leads

### Memorial Pages
- **MMRL-01**: Each case has a public memorial page at `memorial.vigilhq.com/[slug]`
- **MMRL-02**: Memorial page shows photo gallery, guestbook, obituary text

### Review Generation
- **REVW-01**: n8n auto-sends Google review request 14 days post-service via SMS + email
- **REVW-02**: Review link uses tenant's `googleReviewUrl`

### Payment Installments
- **INST-01**: Families can pay over time via Stripe
- **INST-02**: Installment balance tracked per case

## v3 Requirements (Phase 3 — platform maturity)

- **AI-01**: AI obituary drafting via `claude-haiku-4-5`
- **FAITH-01**: Multi-faith task templates per tenant; case `faithTradition` field overrides default checklist
- **MLOC-01**: Multi-location dashboard for ownership groups
- **ANLT-01**: Analytics dashboard (revenue trends, case volume, staff performance)
- **CHAT-01**: AI FAQ chatbot embeddable on client website

## Out of Scope

| Feature | Reason |
|---------|--------|
| Raw card number storage | Legal/PCI risk; Stripe handles processing |
| Custom JWT signing | Cognito owns all credential management |
| BullMQ / Redis job queue | n8n cloud replaces job queue entirely |
| Railway / Supabase / Vercel (prod) | AWS-only deployment commitment |
| NAT Gateway / VPC interface endpoints | Cost ($38/mo+); public subnets + IGW is equivalent with tight security groups |
| Real-time family-staff chat | High complexity; portal covers async communication needs |
| Native iOS/Android apps | PWA sufficient for Phase 2 mobile needs |
| Multi-faith logic in Phase 1 | Phase 3 stub only; not required to sell |
